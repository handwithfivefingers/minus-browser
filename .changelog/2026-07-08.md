# Changelog — 2026-07-08

## Picture-in-Picture crash on YouTube playlist auto-navigation

`onRequestPIP()` in `src/features/tabs/models/tab.ts:424` injected a script with a 500ms `setTimeout` that called `enterPictureInPicture(document.querySelector("video"))`. When YouTube auto-navigated to the next playlist item (SPA), three issues caused an instant crash:

1. **Stale timeout after SPA navigation** — The `setTimeout` survived the SPA transition. When it fired during DOM teardown/build-up, `document.querySelector("video")` returned either `null` or a partially-initialized element, and `requestPictureInPicture()` on a stale element triggered a low-level renderer crash.

2. **Accumulating event listeners** — Each call added new `enterpictureinpicture`/`leavepictureinpicture` listeners without removing previous ones. During navigation, duplicate `leavepictureinpicture` firings resolved the PiP-exit Promise prematurely and created conflicting state.

3. **Unawaited `exitPictureInPicture()`** — `exitPictureInPicture()` was called without `await`, then `requestPictureInPicture()` was called immediately on the same microtask, racing the PiP state machine.

**Fix:** Rewrote the injected preload script (`src/features/tabs/models/tab.ts:21-60`):

- Removed the `setTimeout(500)` delay entirely; operates synchronously on the current video element.
- Added `window.__minus_pip_pending` guard to prevent concurrent/duplicate PiP requests.
- Properly `await`s `exitPictureInPicture()` before calling `requestPictureInPicture()`.
- Re-registers the `leavepictureinpicture` listener after `exitPictureInPicture()` to correctly track the new PiP session.
- Cleans up event listeners via `removeEventListener` on PiP exit.
- Returns `false` for no-ops (already pending, no video found) and a Promise that resolves when PiP actually ends.
